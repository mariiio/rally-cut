# GT resolver canonicalization fix (2026-05-13)

## Bug

`rally_action_ground_truth.resolved_track_id` was carrying stale raw
BoT-SORT ids that no longer existed in the rally's canonical
`positionsJson`. Downstream eval/analytics scripts that join GT on
canonical ids therefore associated each label with the wrong physical
player.

### Root cause

The resolver (`api/src/services/actionGroundTruthResolver.ts`) is correct
on its own — it does bbox IoU + ReID + nearest-center against whatever
candidate set is passed in. The bug was in the **call sites**:

1. `saveTrackingResult` (player tracking write path) invokes the
   resolver against `trackerResult.rawPositions` (raw BoT-SORT ids,
   pre-remap). At that moment the answer is correct relative to raw
   tracking.
2. The match-analysis pipeline subsequently runs `remap-track-ids`,
   which rewrites every per-rally `positionsJson` entry to canonical
   pids (`{1, 2, 3, 4, 101+}`). It does **not** touch
   `rally_action_ground_truth`.
3. Result: `resolved_track_id` is left pointing at the pre-remap raw id,
   while consumers reading `positionsJson` see canonical ids only.

The matchAnalysisService.ts code carried an inline comment ("we
deliberately do NOT canonicalize") justifying the gap by claiming the
web display path would double-translate. That justification was wrong:
the web display path now distinguishes resolver-canonical anchors from
raw `snapshotTrackId` fallbacks, so canonicalization is safe.

## Fix

### Code

1. **`api/src/services/actionGroundTruthService.ts`**
   - Renamed `reresolveRallyGt`'s `rawPositions` parameter to `positions`
     and clarified the docstring: callers should pass canonical
     `positionsJson` (post-remap) so `resolved_track_id` lands in
     canonical id space.
   - Added `reresolveVideoGtAgainstCanonical(videoId)` helper that
     iterates every rally of a video and re-resolves against each
     rally's `positionsJson`.

2. **`api/src/services/matchAnalysisService.ts`**
   - Wired a new stage 4b ("re-resolve GT") immediately after `stage4_remap`
     in both the full-rerun and partial-rerun paths.
   - Updated `applyRemapToRally` (used by single-rally retracks) to
     re-resolve GT against the canonical positionsJson in the same
     transaction as the remap write.
   - Removed the stale "deliberately do NOT canonicalize" comment.

3. **`web/src/utils/gtLabelDisplay.ts`**
   - When `resolvedTrackId` is set AND `resolvedSource` is not
     `UNRESOLVED`, the resolver has run against canonical positions so
     return `resolvedTrackId` directly (no AFM hop).
   - When falling back to `snapshotTrackId` (raw, by definition),
     continue routing through `appliedFullMapping`.

4. **`api/src/scripts/reresolveVideoGt.ts`** — rescue CLI.
   - Walks all GT rows across all (or one) videos, re-resolves against
     canonical `positionsJson`, supports `--dry-run` (default) /
     `--apply`, `--all` / `--video <uuid>`.
   - Reports per-video deltas + aggregate before/after distribution.

### Tests

- `api/tests/actionGroundTruthService.test.ts` adds a test (`Test 8b`)
  that simulates the cascade pattern: save with raw tid=7, simulate
  remap rewriting `positionsJson` to canonical tid=3 (same bbox), call
  `reresolveVideoGtAgainstCanonical`, assert the row's
  `resolvedTrackId` is now 3 with `IOU_MATCH`.
- All 28 GT-related tests pass (resolver + retrack + service).

- 2233 of 2562 rows updated (87%). The remaining 329 unchanged rows are:
  - 304 `UNRESOLVED` (no IoU/ReID/center match available — pre-existing,
    most likely legacy labels at frames where the player wasn't tracked).
  - 25 unchanged because the original guess happened to land on a
    canonical id.

- **SNAPSHOT_EXACT** dropped from 1925 → 207. The 207 residual are all
  rows where `snapshot_bbox_x1 IS NULL` (legacy save-time path couldn't
  derive a bbox), so the resolver fell through to the "trackId equality
  against current positions" branch. Verified: every one of those 207
  `resolved_track_id` values DOES exist in the rally's canonical
  `positions_json`.

- **Sanity invariant**: of 2258 resolved rows in the post-fix table,
  **0** have a `resolved_track_id` that's absent from the rally's
  `positions_json` canonical id set.

## Notes / caveats

- 449 rallies were skipped by the rescue script because their
  `playerTrack.positionsJson` is empty or null (no tracking data); their
  GT rows remain in their prior state. This was 2566 - 2562 = 4 rows in
  null/empty rallies, plus 327 rows whose source was already correct.
  These would land in the same place if/when the rally is tracked.

- The 304 residual `UNRESOLVED` rows are not new contamination — they
  were already `UNRESOLVED` (485 → 304 — net 181 recoveries) or
  decisively unresolvable post-canonicalization. None are silent
  errors.

- The web display contract changed slightly: `resolveGtDisplayPid` now
  trusts `resolvedTrackId` directly when `resolvedSource ≠ UNRESOLVED`.
  Previously it always routed through `appliedFullMapping`. This is
  correct for the new contract; legacy null-resolver rows still fall
  back through AFM via `snapshotTrackId`.
