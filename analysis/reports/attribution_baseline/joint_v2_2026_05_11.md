## STOP VERDICT

**STOP — G-A, G-B, G-C, G-F all FAIL. Do NOT proceed to Task 8.**

Root causes (to investigate):
1. **G-F (10/22 fallbacks, 45%)**: The rule set is too strict for real data. The beam empties
   for nearly half of all rallies. Likely cause: UNKNOWN action types or missing contacts
   break the R1 (first-action-must-be-serve) hard constraint, causing the whole rally to fall
   back. When the fallback fires, joint_attribute returns actions UNCHANGED — but the DB state
   already has v1 team-chain applied, so the OFF/ON states should be identical for those rallies.
   Yet the ON run is WORSE — this means for the rallies where the solver DOES fire (12/22),
   it is actively making wrong assignments.

2. **G-B (cross-team +4)**: The solver fires on 12 rallies, picks rule-valid paths that are
   nevertheless wrong due to soft-scoring failures. The `player_candidates` in DB may have the
   correct-team player ranked behind the wrong-team player, and the hard constraint is not
   helping because the wrong attribution is also rule-valid (e.g., both candidates are on the
   expected team). The serving_team=None seeding from the first SERVE's pid means the solver
   can diverge from ground truth if the first contact is already mis-attributed.

3. **serving_team=None seeding risk**: The `joint_attribute` call passes `serving_team=None`.
   If the pre-existing attribution's first action is wrong (serves mis-attributed to the receiver's
   team), the solver seeds the wrong serving_team and then enforces the wrong team sequence for
   the entire rally — potentially flipping ALL correctly-attributed contacts.

Investigation priority for Task 7 remediation:
- Inspect which 10 rallies fell back and why (what broke the R1 chain).
- For the 12 rallies where solver fired, inspect why cross-team went UP.
- Consider seeding serving_team from DB match_analysis.servingTeam instead of None.
- Consider making R1 a soft constraint (penalty) not a hard constraint when serving_team is unknown.
